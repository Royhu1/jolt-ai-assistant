### Rejected candidates (for audit trail)

- **Row 77 / 2025-03-27 missed overnight charge**: real +51% SOC rise exists in raw
  data but `battery_pack_ac_watthours` column entirely NaN in that window.
  Without charger-type metadata, inventing a Charge Home row is too aggressive.
  Keep as Stop row (user can read the real SOC rise in the figure).
- **Row 198 / 2025-04-21 missed partial overnight charge**: same data-limited
  situation. Algorithm correctly segments the final confident phase as AC Home
  (row 199). The earlier portion stays in the Stop row.

## Confidence assessment

- **Row 352-354 merge**: high confidence. Triangulated via:
  1. Raw telematics timeline showing isolated SOC=44 spike (two samples, 20 s apart)
  2. AC+DC Delta = 0 across the alleged "Charge Home" window (no metered energy)
  3. Total Energy Used delta = 0 across the window (no real consumption flow)
  4. Surrounding SOC values (87 before, 87 after) are consistent with thermal rebalance from fully charged state

### Known library limitation encountered

The `SOC Change (%)` column in merged Stop row 352 still reads `-56` (inherited from
the original first-row value) rather than the correct `-31` implied by Start SOC 100
→ End SOC 69. Inspection of `finetune.py` `_recompute_leg_derived` shows it does not
recompute `SOC Change (%)` for Stop rows (only Duration, and zeros Distance / Speed /
Elevation / EP / Energy Change). This is analogous to the `Energy Change (kWh)`
post-merge staleness documented in the agent gotchas. Recorded here for audit; fix
would need to be routed to `jolt-report-dev`.

Start SOC (100) and End SOC (69) are correct, so any analysis that computes SOC
change from those endpoint columns will produce the right number.
