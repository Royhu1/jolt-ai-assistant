## Stage 2 — Diagnosis of the one real anomaly

### Missed AC charge overnight 2024-09-10 → 2024-09-11

**Evidence from raw telematics (excel_report_database/2.2.2/YK73WFN/raw_telematics/raw_2024-09-{10,11}_*.csv)**:

- 09-10 17:59 Return trip ends @ SOC 21%
- 09-10 19:18:40 `battery_pack_charging_status = CHARGING_AC`, SOC 21%
- 09-10 20:22 SOC 32%, 21:22 SOC 43%, 22:22 SOC 53%, 23:22 SOC 64%
- 09-11 00:22 SOC 74%, 01:22 SOC 83%, 02:22 SOC 93%, 03:08 SOC 100% → `NOT_CHARGING`
- `battery_pack_ac_watthours` cumulative: 9402000 Wh (@ 18:50 09-10) → 9723000 Wh (@ 04:00 09-11) = **+321 kWh AC delivered**

**xlsx row 16** (Stop 09-10 17:59 → 09-12 06:02): SOC 21 → 96, `SOC Change = +75`, `Energy Change = None`.
The 7.9-hour AC Home session is folded into this Stop row. A legitimate segmentation error.

**Root cause (not a finetune concern)**: AC+DC watthours channel in the raw telematics is sparsely
sampled (only ~hourly deltas). The charger-detection layer in `charger_patcher` likely failed to find
a continuous `CHARGING_AC` window that matched the SOC rise tightly enough, so the charge segment was
not emitted. This is a **param-tuner / algorithm-layer issue**, not an xlsx outlier.

**Why NOT applying a SplitOp here**:

1. This is a systematic issue with AC channel sparsity, not a one-off visual outlier — per the
   agent `.md` §Routing Rules it should route to `param-tuner` / `jolt-report-dev`, not be
   patched in finetune.
2. The YK73WFN reference explicitly warns against manufacturing charge segments over flat SOC
   / noisy AC+DC signals. This case differs (SOC +75% is real, not noise), but the fix path
   (two SplitOps: Stop→AC→Stop) would depend on `_apply_split` correctly re-reading raw
   AC watthours into `Energy Charged AC` and the charger-patcher columns — unverified for a
   row originally classified as Stop (with None energy fields).
3. Conservative principle: "the cost of a wrong change >> the cost of a missed change". Creating a Charge row with unreliable `Energy
   Charged AC` / `Wire Energy Efficiency` / `Peak Charging` / `Average Charging` values is
   worse than leaving the SOC-+75 Stop row which at least makes the anomaly discoverable by
   any downstream analysis that inspects SOC_change on Stop rows.
