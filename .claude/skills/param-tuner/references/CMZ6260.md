## Parameter rationale

### `min_stop_duration_min = 15.0`（Round 2 提升 5 → 15）

- 配送停车 5-15 min（红绿灯、过路口、短中转）被吸收，长配送停车 (装卸 ≥ 15 min) 仍切分
- multi-drop city distribution 自然节奏：raw trip plateau 11-13 是 duty cycle 真实状态，不强行压成 5 trip
- Round 2 实测：380 → 341 driving legs (-39, -10.3%)，Issue 天大多从 11-13 trip 降到 8-10 trip

### `trip_endpoint_anchor = zero_speed` + `max_extend_minutes = 5.0`（Round 2 启用）

- 把 trip 端点外扩到最近的 v == 0 telematics 心跳点，避免 trip end 落在 20-40 km/h 的瞬时 dip
- 与 EX74JXW / EX74JXY (scania_speed_00) / 同类配置一致
- Round 1 Issue P2 (~20 cases endpoint v > 0) 已基本消除

### `merge_by_mass = false`（保留 Round 1 设置）

- mass 3 层自然分布稳定 (10 / 18-25 / 35-45 t)，cluster split 准确
- merge 后会把同 cluster 的相邻 trip 合并成长 In Transit，丢失 multi-drop 颗粒度
- 与 KY24LHT / EX74JXW 同策略

### `min_cluster_gap_kg = 1500`（保留）

- mass cluster 自然分布 gap > 5 t，1500 阈值健康；无需调整

## 经验教训

1. **city multi-drop 的 trip 数量天然偏多**：与 KY24LHT 这类 long-haul trunk (1-4 trip/天) 不同；不要强行压缩，否则会丢配送颗粒度。
2. **`min_stop=15` 是 multi-drop 配送的合理 threshold**：5 min 太敏感（红绿灯也切），30 min 太粗（错过短装卸）。15 min 落在装卸时间的下限附近。
3. **`zero_speed` anchor 不直接减 trip 数，但显著提升切分质量**：端点对齐 v == 0 telematics 心跳避免 EP 计算受瞬时速度 dip 干扰。
4. **doubled fig artefact (P3) 跟参数调优无关**：要从 FPS API leg dedup 解决，归 `jolt-toolkit-dev` agent。Param-tuner 报告里只统计 xlsx 实际 leg 数。
5. **R1 评估 doubled fig 天时容易误判**：同日两份 fig 不同切法时，需同时看两份再下"OK"判定。R1 错把 2026-02-16 标"2 trip OK"是只看 _0011 漏看 _0054 的结果。
