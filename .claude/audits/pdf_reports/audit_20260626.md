## Flags (quantified)

### FLAG 1 — EX74JXY: "Total Energy Charged 0 kWh" should be "—" (FAIL, data-truthfulness)
- **Where**: `report_EX74JXY_20250411_20260503.pdf` (+ `_xlsxkpi`), page-1 Charging Sessions card.
  Shows `Total Energy Charged 0 kWh / — AC Charged 0 kWh / — DC Charged 0 kWh` **alongside**
  `No. of Charging Sessions 146`, `Mean Start SoC 52 %`, `Mean End SoC 96 %`.
- **Why it's wrong**: a partner reads "146 charging sessions, SoC climbed 52→96 %, but 0 kWh charged"
  — a self-contradiction. The honest value is "—" (channel not reported), exactly what the SIBLING
  Scania EX74JXW correctly shows.
- **Root cause (verified in raw + xlsx + code)**: EX74JXY has NO raw charge/energy counters — all of
  `battery_pack_dc/ac_watthours`, `total_electric_energy_used`, `electric_energy_recuperation_watthours`
  are present-but-100%-NaN across all 360 raw files (49,082 rows, 0 non-null) — so the raw basis
  correctly falls back to segment. The "0" comes from the **xlsx event charge legs**: 20 of 146 charge
  legs carry a genuine `0.0` (not NaN) in `Energy Charged AC (kWh)` and `Energy Charged DC (kWh)`
  (EX74JXW has 0/62 non-null → all NaN). `_sum_or_na(s)` returns `s.sum()` whenever `s.notna().any()`
  is True → for EX74JXY it returns `0.0` (a "real" zero), so the render shows `0 kWh` instead of `—`.
- **Second symptom (same cause)**: because `tot_ch = 0` (not NaN), `pd.isna(tot_ch)` is False, so the
  page-1 data-availability note for EX74JXY reads only *"energy recuperated are not reported…"* and
  **omits "charged energy (AC/DC)"** — whereas EX74JXW correctly lists *"charged energy (AC/DC) and
  energy recuperated"*. So EX74JXY both shows a false 0 and fails to disclose the channel as missing.
- **Proposed fix** (route to `generate-pdf-report` skill — owns the KPI computation): treat a charge
  AC/DC channel that is **present-but-all-zero** (no positive value across all charge legs) as
  unavailable → `—`, e.g. in `_sum_or_na` (or a charge-specific guard) return NaN when
  `(s.fillna(0) == 0).all()` AND there is no raw counter. Optionally the underlying spurious `0.0`
  charge-energies on a no-charge-counter Scania could be suppressed upstream (jolt-toolkit), but the
  display guard in the skill is the safer, self-contained fix. Verify EX74JXW stays "—" after.

### FLAG 2 — T88RNW: page-1 Summary overflows the page (FAIL, layout)
- **Where**: `report_T88RNW_20240611_20260623.pdf` (+ `_xlsxkpi`), page 1.
- **Evidence (coordinate-based, not by eye)**: A4 page height = 841.92 pt. The Summary's final
  wrapped line `"—").` has bbox y0=829, **y1=845 pt — 3 pt below the page bottom edge** (every other
  briefing maxes at y1≈798). The bottom of the last glyph row is clipped.
- **Root cause**: T88RNW is the **only** no-mass (distribution-variant) vehicle that **also reports
  regen** (it has the Renault recuperation counter). So its page-1 Summary carries **4** bullets —
  fleet line + charging line + regen-recovery line + data-availability note (GVM missing) — one more
  than the other no-mass vehicles (YN75NMA/LN25NKE have no regen line → 3 bullets, y1≈773) and one
  more than the standard vehicles' 3-bullet summary. The 4th bullet pushes the card past the 297 mm
  page bottom (`.page { overflow:hidden }` then clips it).
- **Proposed fix** (route to `generate-pdf-report` skill): tighten the page-1 Summary block for the
  4-bullet case (slightly smaller line-height / margin, or fold the regen line into the fleet line,
  or shorten the data-availability note) so the card fits within the page for a no-mass + regen
  vehicle. Re-run the coordinate check (max page-1 y1 < ~828 pt) after.
