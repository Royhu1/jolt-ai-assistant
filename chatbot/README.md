# chatbot/ — JOLT Fleet Assistant

> A **web chatbot** dedicated to answering questions about the JOLT battery-electric HGV
> research project: a vehicle's specs / energy use, the energy difference between two
> vehicles, a fleet overview, terminology, and so on. The UI has clickable example
> questions, multi-session history, new-chat, and a **bilingual EN / 中 toggle** (default
> English; the top-right `EN / 中` switch flips the UI text, example questions, offline
> rule-engine answers, and the online Claude answer language together).
>
> The **Appearance** panel at the bottom of the sidebar toggles a **Light / Dark theme**
> and adjusts the reading-area **font size** with `A− / A+` (default slightly large).
> Theme, font size and language persist in `localStorage`.

> Powered by your existing **Claude Code subscription** — **no API key, no per-token
> billing**. When you don't want to spend quota, the page also works **fully offline** via
> a built-in rule engine for example-type questions.

## Two ways to use it

### Mode A — online Claude subscription (recommended; ask anything)

Start a local bridge service that forwards your question to a headless `claude -p` (on the
subscription) and returns the answer to the page:

```powershell
python chatbot/bridge.py
```

Or double-click **`chatbot/start.cmd`** (Windows) / run **`chatbot/start.ps1`**. The service
opens <http://127.0.0.1:8765/> in the browser automatically. Switch the top-right control to
**"⚡ Claude subscription"** and ask anything.

Common options:

```powershell
python chatbot/bridge.py --port 9000 --model opus --no-open
```

### Mode B — fully offline (no network, no quota)

Just **double-click `chatbot/index.html`** to open it in a browser — no service needed. The
built-in rule engine answers:

- basic / detailed info for a vehicle (e.g. `YK73WFN`, `CMZ6260`)
- energy comparison between two vehicles (e.g. "AV24LXJ vs AV24LXK")
- fleet overview / most efficient / OEM comparison
- terminology (EP, SOC, regenerative braking, cruise correction …)

Offline mode does not depend on the bridge and never consumes subscription quota.

## Billing notes (important)

| Situation | Note |
|-----------|------|
| **Now (≤ 2026-06-15)** | `claude -p` on the subscription counts against your **shared usage quota**, not billed per token |
| **From 2026-06-15** | `claude -p` / Agent SDK usage moves to a **separate monthly quota** (Pro $20 / Max 5× $100 / Max 20× $200), needing a one-time opt-in on the account; still inside the subscription, not pay-per-use |
| **The only way it bills the API** | an `ANTHROPIC_API_KEY` set in the environment → Claude Code switches to API-rate billing |

**Structural safeguard** (forces the subscription path; does not rely on "you happening not
to have set a key"):

- when invoking `claude.exe`, **strip every environment variable** that could redirect to a
  non-subscription / metered path: `ANTHROPIC_API_KEY` / `ANTHROPIC_AUTH_TOKEN` /
  `ANTHROPIC_BASE_URL` / `ANTHROPIC_API_URL` / `ANTHROPIC_CUSTOM_HEADERS` /
  `CLAUDE_CODE_API_KEY_HELPER` / `CLAUDE_CODE_USE_BEDROCK|VERTEX|FOUNDRY` /
  `AWS_BEARER_TOKEN_BEDROCK`.
- pass `--setting-sources ""` — **load no settings.json**, ignoring any `apiKeyHelper`
  configured there (which would otherwise re-inject an API key after the env strip).
- print which variables were actually detected and stripped at startup.

## Data source and updates

The knowledge base `data/fleet_kb.json` (and the page's `fleet_kb.js`) is generated by
`build_kb.py` from the project's real **aggregated data** (no raw telematics, no keys — safe
to commit):

| Source | Use |
|--------|-----|
| `src/jolt_toolkit/configs/vehicles.json` | vehicle specs (capacity, battery, model, torque) |
| `test_data_config.json` | data-coverage date ranges |
| `data_analysis_workspace/ep_multifactor_regression/data/fleet_trip.csv` | 17k-row per-trip energy aggregation |
| `data_analysis_workspace/.../cross_vehicle/summary_table.csv` | per-vehicle regression + EP stats |
| `data_analysis_workspace/driving_cycle_correction/results/telematics_fcycle_fleet_summary.csv` | cruise-correction EP@90 |

**Re-run when the data changes** (e.g. new reports generated, a new vehicle onboarded):

```powershell
python chatbot/build_kb.py
```

Current coverage: 15 vehicles (14 BEV + 1 diesel, 5 OEMs), of which 11 have per-trip energy
stats; ~17,063 trips, 437,605 km. The diesel `WU70GLV` is measured in L/100km and is not
directly comparable to BEV kWh/km. Recently-onboarded, under-sampled vehicles (`CMZ6260` /
`T88RNW` / `YN75NMA`) currently have spec info only.

## How it works

```
browser ──POST /api/chat {message, session_id}──▶ bridge.py
                                                   │ strip all billing-related env vars
                                                   │ cwd = neutral temp dir (no CLAUDE.md)
                                                   ▼
                              claude.exe -p --output-format json
                                  --setting-sources ""      ← ignore apiKeyHelper
                                  --tools ""                ← all tools off
                                  --strict-mcp-config       ← no user MCP
                                  --disable-slash-commands  ← no skills
                                  --system-prompt-file <KB injected>
                                  --resume <session_id>     ← keep multi-turn context
                                                   │
                          {result, session_id} ◀──┘
browser ◀──{reply, session_id}── bridge.py
```

- **Session continuity**: each chat tab keeps a `session_id`; follow-ups use `--resume` to
  retain context; "new chat" = a fresh session. Sessions are persisted by Claude Code, so they
  survive a bridge restart.
- **Sandboxing**: claude runs in a neutral temp dir with **all tools / skills / user MCP
  disabled**, answering only from the injected KB — it never touches your repository.
- **Local-only**: the service binds `127.0.0.1` by default, validates Host (anti DNS-rebinding)
  and Origin (so other web pages can't cross-site-borrow your quota), and does not serve `.py`
  source; a non-loopback address requires an explicit `--allow-lan`.

## FAQ

- **The page is stuck in "offline rule mode"** → the bridge isn't running, or the port is
  taken. Run `python chatbot/bridge.py`, confirm the console printed the URL; use `--port` on
  a clash.
- **"claude not found"** → confirm Claude Code is installed (`claude --version`) and
  `claude.exe` is in `~/.local/bin/` or on PATH.
- **The numbers in answers aren't current** → re-run `python chatbot/build_kb.py` to refresh
  the KB.
- **Want a stronger model** → `--model opus` (more quota); the default `sonnet` is more
  cost-effective.
