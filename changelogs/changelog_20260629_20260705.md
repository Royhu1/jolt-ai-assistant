# Changelog 2026-06-29 — 2026-07-05

## Q: Fix the data-availability dashboard operator overlay showing EX74JXW as "unknown" from 2026-01 onward (it is operated by WELCH_TRANSPORT from Feb 2026)

**Task**: On `excel_report_database/2.2.6/dashboard/data_dashboard.html`, EX74JXW
showed operator "unknown" for 2026-02→2026-04. Root cause (pre-diagnosed): the
dashboard operator overlay was built **only** from the manually-curated
`plot_config.json` `company_assignment`, whose `round_robin.EX74JXW` listed only
DP_WORLD (…2025-08) and WJF (…2025-12) — the WELCH_TRANSPORT period (2026+) was
never added, so dates after 2025-12-31 matched no period → "unknown". The xlsx
reports' `Operator` column (SRF `leg.trip.trial.description`-derived) already held
the correct value.

**Result**: Made the dashboard operator overlay **data-driven from the reports'
`Operator` column**, aligning it with how the report generator and PDF briefing
resolve operator. Changes (all in `src/jolt_toolkit/report_generator/`):

- `data_dashboard.py`:
  - `read_report_legs()` now reads the `Operator` column per leg (looked up via
    the live header row with `idx.get`, so older artefacts without the column
    degrade gracefully); `_collect_legs_by_reg()` backfills the operator across
    de-duplicated legs.
  - New `_operator_days_from_legs()` collapses a vehicle's per-leg operators to
    one operator per day (daily majority); `_periods_from_operator_days()` groups
    consecutive same-operator days into dated periods — a single distinct
    operator ⇒ "Fixed" (one open-ended period), ≥2 ⇒ "Round-robin" (one dated
    period per run). Blank/`nan` operator legs are ignored.
  - `build_operator_assignment(plot_cfg, op_days_by_reg=None)` rewritten:
    data-driven periods are primary; the manual `company_assignment` is kept only
    as a **fallback** for a vehicle with no operator in the data (nothing
    regresses). Operators absent from `colors.company` (e.g. `HTL`) get a stable,
    deterministic fallback colour (`_stable_operator_colour`, md5-indexed
    palette). `_prettify_company` simplified to plain underscore→space (case
    preserved: `WELCH_TRANSPORT` → "WELCH TRANSPORT"), matching the briefing.
  - `scan_report_database_full()` stashes per-vehicle `operator_days`;
    `render_dashboard_html()` feeds them into `build_operator_assignment`.
- `data_dashboard_detail.py`: the drill-down detail pages share the same
  data-driven operator assignment (built from the same per-day operator map).
- `src/jolt_toolkit/README.md`: documented the data-driven operator-overlay
  source.

**Config fallback kept**: yes — `plot_config.json` `company_assignment` is still
consulted, but only for a vehicle whose reports carry no operator at all. (For the
current 2.2.6 database every vehicle has populated operator data, so the fallback
is dormant.)

**Q: report_EX74JXW_DP_WORLD 的 EP 随温度正相关(与其他车相反),找原因;并改进:正相关时放宽质量窗口直到负相关,实在不行改文字说明。**
A: 根因:**伪相关、非真实现象**。简报温度图只用"laden 簇里的窄质量窗口"(控质量)拟合;EX74JXW DP_WORLD 是**纯夏季窗口**(2025-07~08,11–30°C,无冷天→缺制热主导信号),且温度对 EP 本就极弱(R²≈0.03)。窄窗(24.6–26.6t,n=34)斜率偶然+0.0075、R²0.03(纯噪声);而**全部行程斜率 −0.011(负、与各车同向)**。其他车是 `--all-data` 跨深冬(−6°C)→负向明显。**改进**(skill 自有代码):`_compute_load_points` 改用新 `_adaptive_temp_window`——窄窗正斜率时逐步放宽质量窗(2→3→4→5t→整个 laden 簇)**直到 EP-温度斜率≤0**;若整簇仍正→`temp_status="inconclusive"`,结论改文字"本周期温度范围不足以判定温度影响"+`_scatter_fit(fit=False)` 不画拟合线。验证:DP_WORLD 已变 "(laden trips, 23–27 t): … per 10 °C **colder**"(窗宽4,斜率−0.0035);WJF 等本就负向不受影响。全队重生成中。改动未提交。
