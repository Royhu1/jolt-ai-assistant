## 2026-04-13

### Q: 修复 SRF Logger 布尔字段丢失（CC active / brake / clutch），版本 bump 到 2.2.1

**背景**：`analysis/EP_cruise_report.md` 的 cruise 识别使用速度 + 加速度阈值器，
用户怀疑 SRF Logger 数据里本来应该有 `CCVS cruise control active`，
但项目代码在拉取时把它丢掉了。

**调查过程**（纯证据链）：

1. 扫描 10 辆车全部 raw_logger_v1/v2 CSV（超过 100 万行），确认 `CCVS cruise control active`、
   `brake switch`、`clutch switch` 三列**全部为空**；只有 EX74JXW 的 `set speed` 有 99.99% 覆盖
2. 顺着 `srf_client/pandas.py` 查到 `to_numeric`：它只尝试 `int` → `float`，遇到字符串抛 NaN
3. 本地验证 `pd.DataFrame.from_records` 对短 tuple **会抛 ValueError 而非补 NaN**，
   证明 `m.data` tuple 长度必定等于 schema fields 长度（5）
4. 用户提供手动下载的 YK73 raw measurement CSV（`20250830092649_BCD11F1E136D.csv`），
   肉眼可见 CCVS measurement 格式为 `64.31640625,NaN,false,false,false` —
   **5 个字段，后 3 个是字符串 `"false"/"true"`**
5. 根因确认：`to_numeric("false")` → `float("false")` ValueError → `math.nan`。
   `srf_client` 的默认 conversion 不认识布尔字符串，所有布尔 SPN 丢失

**修复**：

1. 在 `src/jolt_toolkit/report_generator/_generator.py` 新增 `_logger_to_numeric`，
   是 `to_numeric` 的**严格超集**：在原逻辑前加 `"true"→1`、`"false"→0` 映射
2. `_save_logger_data` 调用 `leg.get_data_frame(..., conversion=_logger_to_numeric)`
3. 离线回放用户手动 CSV：CCVS 的 5 个字段 non-null 率从 `(100, 0, 0, 0, 0)%` → `(100, 0, 100, 100, 100)%`，
   `active` unique 值 = `[0, 1]` ✓
4. 在线端到端跑 YK73 (2025-06-01 ~ 2025-09-01)：159 条 trip / 486 456 行，
   **CC active 非空率 99.0%**，`active==1` 占 19.92%（全局）/38.8% 的 trip 至少用过一次 CC，
   平均每 trip CC-on 时间 ~10.6 分钟 ✓
5. `brake switch` 同时得到 0/1 两种值 ✓

**产出**：

- `_generator.py`：新增 `_logger_to_numeric` 函数，改写 `_save_logger_data` 传入 conversion
- `pyproject.toml`：version 2.1.0-dev → **2.2.1**
- `src/jolt_toolkit/README.md`：更新包结构版本号、展开 CCVS 列清单、新增"布尔字段转换（v2.2.1 修复）"小节
- git tag v2.2.1

**影响**：

- `logger_patcher.py` 的 `_load_speed_data` 只读 wheel speed（浮点），不受影响
- 现在可以在 analysis 侧用真实 `cruise control active` 位作为 cruising segment 的判据，
  兑现 `simulation/EP_simulation_report.md` §1.5.7 Case 1 的零模型参数测量
